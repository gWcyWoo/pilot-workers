# Provider and Isolation Contract

## Locked Routes

| Skill provider | OpenCode provider ID | Exact model | Context / output limits | Official base URL |
| --- | --- | --- | --- | --- |
| `glm` | `glm-worker` | `glm-5.2` | 1,000,000 / 131,072 | `https://open.bigmodel.cn/api/coding/paas/v4` |
| `kimi-k3` | `kimi-worker` | `k3` | 1,048,576 / 1,048,576 | `https://api.kimi.com/coding/v1` |

Both routes use OpenCode's `@ai-sdk/openai-compatible` Chat Completions adapter. Do not accept a provider ID, model ID, base URL, or adapter override from a task.

## Runtime Contract

- OpenCode package: `opencode-ai@1.18.3`.
- Install root: `$CODEX_HOME/worker-runtime/opencode/1.18.3` (default `$HOME/.codex/...`).
- Invocation: pinned binary with `--pure`, non-interactive `run`, JSON output, exact `--model`, and exact mode agent. The task contract is fed via stdin (OpenCode merges non-TTY stdin into the run message); it never appears in argv or the process list.
- Runtime configuration: `OPENCODE_CONFIG_CONTENT` generated by the runner and applied at highest documented precedence.
- Provider allowlist: the selected custom provider only.
- Main and small model: the same locked provider/model.
- Sharing: `disabled`; never pass `--share`.

OpenCode documents inline configuration precedence, exact `provider/model` selection, custom OpenAI-compatible providers, headless `run`, JSON events, and permission denials:

## Credential Contract

Each provider gets its own XDG tree under `$CODEX_HOME/opencode-workers/providers/<provider>/`:

- `config/`
- `data/opencode/auth.json`
- `state/`
- `cache/`

The credential file contains only that profile's custom provider entry and uses mode `0600`; profile directories use `0700`. The runner does not export a GLM, Kimi, Anthropic, OpenAI, or relay API key. It passes a sanitized environment to OpenCode and redacts the selected provider key if it unexpectedly appears in child output.

The key necessarily travels as authorization over HTTPS to the selected official provider. It is not sent to a relay. The task and relevant code context also travel to that provider, so dispatch only approved source material.

## Claude and Plugin Separation

The runner sets:

- `OPENCODE_DISABLE_CLAUDE_CODE=1`
- `OPENCODE_DISABLE_CLAUDE_CODE_PROMPT=1`
- `OPENCODE_DISABLE_CLAUDE_CODE_SKILLS=1`
- `OPENCODE_DISABLE_DEFAULT_PLUGINS=1`
- `OPENCODE_DISABLE_LSP_DOWNLOAD=1`
- `OPENCODE_DISABLE_MODELS_FETCH=1`
- `OPENCODE_DISABLE_AUTOUPDATE=1`
- `OPENCODE_AUTO_SHARE=false`

It also invokes `--pure`, denies skills, subagents, MCP tools, web tools, external-directory access, remote Git operations, network shell clients, credential-file paths, and destructive root/home deletion commands. Never reuse a Claude XDG tree, runner, prompt, session, skill directory, or auth file.

Shell permission semantics (verified against the pinned 1.18.3 binary and by live probes): OpenCode parses each bash command with tree-sitter and checks every command node in the AST separately — pipeline segments, `&&`/`;` list members, and commands nested in `$(...)` each hit the allow/deny rules on their own. Operator-symbol deny rules are therefore not used (they would never match a segment and would false-positive on quoted arguments). The one exception: redirected statements are matched with their full text, so read-only modes keep a `*>*` deny to stop redirect writes from riding a prefix allow rule like `rg *`.

These controls isolate provider profiles and reduce accidental or prompt-driven cross-access; they are not an operating-system security boundary. OpenCode must be able to read its own credential, and it runs under the current user. Do not use this runner for hostile repositories or adversarial task text. If the worker itself is outside the trust boundary, add an OS/container sandbox or a narrowly scoped credential broker before dispatching it.
